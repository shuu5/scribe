00

TL;DR — 3行でいうと

SUMMARY
WHAT

コードが制御する多エージェント

dynamic workflow = JavaScript スクリプトで多数のサブエージェントを決定論的に指揮する仕組み。ループや分岐は LLM の気分でなく for/while が決める。コードは繰り返しを忘れない。

WHY

速度でなく網羅性・確信度を買う

並列化の本質は時短ではない。互いに盲目な独立エージェントの多視点で「もっともらしいが間違い」を潰し、1コンテキストに収まらない量を制覇する。そのための定型パターン集がある。

SCRIBE

背骨にせず、道具に限定

scribe は dynamic workflow を組織図（背骨）にしない。worker はあくまで完全な対話セッションで、WF はタスク内の bounded な戦術 fan-out 専用。凍結骨格2本＋args 注入で再利用する。

01

基本知識 — 何が「dynamic」なのか

FUNDAMENTALS

1.1 制御フローを誰が握るか — Agent tool との対比

Claude Code でサブエージェント（＝親とは別のコンテキストで動く子エージェント）を使う道は2つ。違いは「次に何をするか」を誰が決めるかに尽きる。

AGENT TOOL — LLM 駆動
親 LLMその都度、判断して spawn subagent ×1
  • 単発の調査・独立作業 1 個に向く
  • 「10回繰り返して」は途中で忘れうる
  • fan-out の形が実行のたびに揺れる
WORKFLOW TOOL — コード駆動（dynamic workflow）
scriptfor / while / if / schema 検証 agentagentagent…×N
  • ループ・条件分岐・大規模 fan-out に向く
  • 決定論的＝同じ入力なら同じ流れ。コードは忘れない
  • 途中失敗からの resume（キャッシュ再開）が可能

1.2 スクリプトの解剖 — 5つの道具で全部読める

find-bugs.workflow.js — 最小の骨格
export const meta = {                      // 必ず先頭・純リテラル（変数や計算式は書けない）1
  name: 'find-bugs',
  description: 'バグを探して検証する',
  phases: [{ title: 'Find' }, { title: 'Verify' }],
}
phase('Find')                                  // 進捗表示のグループ開始2
const found = await agent('この diff のバグを探せ', {
  schema: BUGS_SCHEMA,                          // 返り値の型を JSON Schema で強制3
  model: 'opus', label: 'find:bugs',
})
log(`${found.bugs.length} 件検出`)             // ユーザーに見える進捗 1 行4
const verdicts = await parallel(found.bugs.map(b => () =>
  agent(`反証せよ: ${b.title}`, { schema: VERDICT, phase: 'Verify' })))5
return { confirmed: verdicts.filter(Boolean).filter(v => !v.refuted) }
1meta — 名前・説明・フェーズ一覧。純リテラル制約（run ごとに変えられない）。これが後述の「run 識別問題」の伏線になる。
2phase() / log() — 進捗の可視化。phase タイトルは meta と完全一致でグループ化される。
3agent(prompt, opts) — サブエージェント1体。schema（＝返り値の型定義）を渡すと検証済みオブジェクトが返る（型不一致は自動リトライ）。「子の返事を親がパースする」壊れやすさが消える、実務上いちばん重要な機能。
4args（この例には無いが）— 呼び出し時に外から渡す入力。骨格の再利用の鍵（§2.4）。
5parallel() / pipeline() — 並列実行の2形態。使い分けが次項。

1.3 pipeline vs barrier — 待ち合わせの規律

parallel() は全員終わるまで待つ（この待ち合わせ点を barrier と呼ぶ）。pipeline() は barrier を張らず、各アイテムが自分のペースで次の段へ進む。多段処理の既定は pipeline。

BARRIER 型（parallel を2回）— 速い子が最遅の子を待たされる
item A
stage1stage2
item B
stage1（最遅）stage2
item C
stage1stage2
PIPELINE 型（既定）— 各自、終わり次第すぐ次の段へ。A は 5、C は 4 の時点で完了
item A
stage1stage2
item B
stage1stage2
item C
stage1stage2
stage1（例: review）stage2（例: verify）barrier 待ちの無駄
barrier が正当化されるのは全結果を一度に必要とするときだけ — 重複排除・0件なら後段スキップ・相互参照。「概念的に段が分かれているから」は理由にならない（それは pipeline が表現するもの）。

1.4 実行環境の重要な制約

同時実行 cap

同時に走るのは min(16, CPUコア−2) 体まで。100体渡しても超過分は待ち行列に入って全部完走する。生涯上限は1000体（暴走防止の天井）。

Date.now() 禁止

Date.now() / Math.random() は使えない。理由は resume: 中断後に同じスクリプトで再起動すると変更のない agent() はキャッシュから即返る。実行のたびに値が変わる関数は再現を壊す。

バックグラウンド実行

ワークフローは裏で走り、完了すると親に通知が届く。親はその間、別の作業を続けられる（このページの元ネタ収集もそうやって走った）。

02

応用知識 — 何を買うために fan-out するか

QUALITY PATTERNS

2.1 発想の転換

fan-out（＝並列展開）の価値は時短だけではない。買っているのは次の3つ。

網羅性

1つのコンテキストに入りきらない量を、分割して全部読む・全部触る。

確信度

互いの結果を知らない独立エージェントに別角度から検証させ、1体の「もっともらしいが間違い」を潰す。

スケール

移行・監査・全数スイープ — 1コンテキストでは持ちきれない作業量に挑む。

2.2 品質パターン・カタログ（7つの語彙）

出典: scribe docs/methodology.md §2。各パターンは「どの軸に効くか」のタグ付き。

不確実性に効く

adversarial verify（敵対的検証）

finding（指摘）1件ごとに独立した懐疑役を立て「反証せよ。迷ったら却下」と命じ、多数決で殺す。誤検出（plausible-but-wrong）を本線に通さない。

finding→skeptic×3→生存/棄却
不確実性に効く

perspective-diverse verify（多視点検証）

検証役 N 体を同一の複製でなく別レンズ（正しさ／セキュリティ／再現性／性能）にする。冗長より多様のほうが広い壊れ方を拾う。

correctnesssecurityreproperf
規模（広い解空間）に効く

judge panel（審査員団）

設計案を別角度（MVP優先／リスク優先／ユーザー優先）で N 本独立生成→並列採点→勝者に次点の長所を移植。一発生成の反復より広い解空間を当てる。

不確実性（未知サイズの発見）に効く

loop-until-dry（枯れるまで回す）

バグ等個数が読めない発見は「K 回連続で新規ゼロ」まで finder を回す。count < N 型は尻尾を取りこぼす。重複排除は「確定リスト」でなく「見たことある集合（seen）」に対して行う — でないと却下済みが毎回再来して永遠に収束しない。

規模（探索の広さ）に効く

multi-modal sweep（多角スイープ）

1つの検索角度で拾えないものを、互いに盲目な複数角度（内容別・実体別・時系列別…）で並列に掃く。

リスク（見落とし）に効く

completeness critic（完全性批評家）

最終段に「何が欠けているか — 走らせていない角度・未検証の主張・未読のソース」だけを問う批評役を置く。出たものが次ラウンドの仕事になる。

リスク（誠実性）に効く

no silent caps（黙った切り捨て禁止）

top-N・リトライなし・サンプリングで網羅を切ったら、必ず log() で何を落としたかを明示する。黙った切り捨ては「全部見た」と誤読される。

2.3 強度キャリブレーション — 全部入り≠最善

fan-out の本数・検証の票数・ループ回数はすべて実トークンとレート枠を消費する。強度は3軸で見積もり、効く軸にだけ盛る（methodology.md §1）。

3軸 → 上げるべきレバー
規模触る面積・読む量──▶fan-out の幅finder 数 / item 数 / 角度の数
不確実性解の未知さ・誤検出の出やすさ──▶検証の深さ1票 → 3〜5票 反証多数決 / loop 回数
リスク不可逆性・外部公開・波及範囲──▶gate の厳しさread-only 監査 / 人間確認の閾値
コスト実測（verified）: 2026-06-10 のパイロットでは gate review 2回で 5時間レート枠を 53%→80% まで消費（fable 投入起因の特殊コスト。これが後述「WF から fable 全廃」の根拠になった数値）。強度選び＝レート予算の配分そのもの。「ユーザーが要求した強度」が上限で、opt-in なしに勝手に盛らない。

2.4 骨格の凍結と再利用 — args という差し込み口

よくできたスクリプトは資産。汎用の骨格を ~/.claude/workflows/*.workflow.js に凍結し、案件ごとの固有物は args で注入する。同じ骨格を二度書かない。

骨格 × args — 不変ロジックと固有データの分離
凍結骨格フェーズ順序・ループ・安全装置
= git 管理・全 project 共有 ◀── args ── 固有データworktree / テストコマンド /
観点 / 対象パス

運用則①: まず ls

新規に組む前に ls ~/.claude/workflows/。カタログはコンテキストに自動注入されないため、これを踏まないと既存骨格に気づかず再発明する。

運用則②: 起動直後に log で run 識別

meta.name は純リテラル制約で run ごとに変えられない → 同名 WF を並走させると進捗ビューで区別不能。最初の1行で log('[タスク名] …') を出すのが唯一の識別手段。

scribe での仕組み — 背骨にせず、道具にする

SCRIBE INTERNALS

3.1 大前提: 2層モデル（最重要の設計判断）

scribe は dynamic workflow をオーケストレーションの背骨にしない（scribe-design.md §12・§18.0）。理由: workflow の子エージェントは親のターンと共に死に、フラットで、途中入力ができない。持続・並列・相互通信する worker 艦隊の土台にならない。

背骨層 — 確定済み・設計余地なし（組織図）
adminanchor セッション
graph 所有・gate・唯一の dolt push ── spawn ──▶
worker ×N完全な対話 CC セッション
1 issue = 1 worktree = 1 window
worker 自身が実装する
↑ ここ（タスク内）でだけ WF を使う
beads (bd)タスク台帳・依存 DAG
= 調整の SSOT

オーケストレーションの SSOT = folio spec + admin プロトコル + hook ゲート + beads。workflow スクリプトではない。

戦術層 — 設計余地あり・D1-D7 で確定（電動工具）

worker や admin がタスク内の bounded な fan-out（品質レビュー・調査スイープ・pre-bake 分析）に限って dynamic workflow を使う。凍結骨格は2本だけ:

3.2 凍結骨格①: cell-quality — 実装セルの品質保証ループ

「1 issue = 1 実装セル」の品質を担保する戦術層の主役。§2 のパターンがほぼ全部入り。WF 自身は合否を確定せず、構造化 JSON を返すだけ（薄 gate 設計・後述）。

CELL-QUALITY PIPELINE — loop-until-dry で収束するまで回る

タスク種別を判定（testable / executable / docs / config / monitoring / notes）。種別ごとに検証戦略が変わる（テスト系は TDD red→green、実行系は dry-run・実機実証…）。軽量種別（monitoring/notes）はループ1周のみ。
[PLAN]
[IMPLEMENT]任意・opus

REVIEWperspective-diverse

correctnessrobustness-securityintegration-opscompleteness-critic

指摘1件ごとに独立の検証者。プロンプトで反証バイアス:「立証責任は指摘側。少しでも不確かなら refuted=true」。生き残った critical/major だけ（confirmed blocking）がループを駆動し、minor/nit は記録のみ。
※ 観点間は pipeline（correctness の review が終わり次第その verify が走る）、観点内の複数指摘は parallel — §1.3 の規律の実例。

CONVERGED（収束）

critical/major が 2ラウンド連続ゼロ（zeroStreak≥2）で初めて収束。レビューは非決定的なので1回のクリーンを信用しない。

OPEN（続行）

blocking が残り autoFix 不可なら、呼び出し元が修正して再起動する番。

ESCALATE（人間へ）

ループ上限到達・self-test 失敗・機構故障（review/snapshot の失敗はクリーン扱いしない）→ 人間判断に差し戻し。

骨格本体にはラッパーから上書きできない不変条件が多重防御としてハードコードされている（D4「安全の核は内」）:

demoteFable — fable の機械的降格

review/verify のモデルが fable に解決されたら問答無用で opus へ降格。fable は実コスト 2×Opus 超 → dynamic WF への投入は全廃（2026-06-10 改訂）というルールをコードで強制。FABLE_MAX_CONCURRENCY=2 は降格漏れ時の最終防壁。

makeLimiter / opusLimiter — セマフォ

セマフォ（＝同時実行数を絞る待ち行列）。args.maxConcurrency で opus 並列度を案件ごとに絞れる（D2: レート予算は案件ごとに違うから args 化）。

receivedArgs エコー

「どんな args が届いたか（キー一覧＋型）」を全 return 経路で返す。args がシリアライズ事故で全既定値実行に落ちると偽クリーンになる — 呼び出し元が机上でなく値で監査できるようにする。

defensive parse / fail-fast

args の JSON 解釈失敗・必須 args 欠落時はエージェントを1体も起動せず escalate で即 return。静かな暴走を根治。autoFix も既定 off（共有 fail-safe）。

3.6 D1–D7 — 戦術層の設計判断（決定と why）

2026-06-15 の grill で確定（scribe-design.md §18）。クリックで why を展開。

D1戦術層の背骨 = ハイブリッド骨格 + args
決定: 不変骨格（cell-quality.workflow.js）＋ worker が args を供給。WHY骨格を毎回書き下ろすと収束硬化や read-only 不変条件が自由記述に溶けて再現性を失う。全部を骨格に焼くと固有物のたび分岐が増える。不変な品質構造は骨格・固有物は args が再現性と柔軟性を両取りする。
D2opus 並列 cap = args 化maxConcurrency
決定: A.maxConcurrency で opus 経路の同時実行を絞れるようにする。WHYレート予算は案件ごとに違うから、ハードコードでなく args。既存の makeLimiter セマフォに opus 経路を通すだけ＝新規機構ゼロ。
D3dimensions 権限 = 枠分業床は admin / 上積みは worker
決定: admin gate は必須4観点固定、worker 自己点検は 4必須＋focus 調整＋追加観点可。WHYgate は一次監査の最低保証ゆえ観点が揺れてはいけない。worker はタスクを一番知る主体ゆえ調整できるほうが検出力が上がる。完全固定も完全自由も両端却下。
D4固定/可変の境界 = 両方安全の核は内・固有合成は外
決定: 絶対不変（read-only 強制・収束硬化・demoteFable）は WF 本体にハードコード、固有合成は外部ラッパー（gate-args 型）。WHY一次監査の安全条件はラッパーから上書きされては困るので骨格内に固定（fail-safe を外部化しない）。固有合成を外に出せば骨格を触らずに済む。改竄耐性と再利用性の分離。
D5汎用/固有の写像ロジックは汎用・固有はデータだけ
決定: 骨格＋汎用ラッパー＝scribe（全 project 共有）、固有「データ」（selfTestCmd・追加観点・probe）＝project 側。WHY固有データをロジックへ混ぜると project ごとに WF が fork して保守が割れる。データを外から注げば汎用ロジック1本で全 project を回せる。
D6per-project WF ファイル = 不要YAGNI
決定: 汎用1本＋args/ラッパーで吸収。骨格の形（phase 順序・ループ構造）自体が違う要件が現れたら、その時に作る留保。WHYper-project ファイルは骨格の重複＝ドリフト源。まだ現れていないものは作らない。
D7compaction 対応 = 参照保持carrier を増やさない
決定: run 固有の WF 呼び出し（scriptPath + args）は Working Memory に参照として残すだけ。骨格は git、恒久設定は CLAUDE.md という既存 carrier に乗せる。WHYrun 固有の呼び出しは「その run 限りの一時状態」なので Working Memory が正しい置き場。新しい保存先を作ると責務境界が壊れる。
