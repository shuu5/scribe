# scribe/scripts

scribe plugin の道具（手順をコード化する薄いヘルパー群）と hook script を置くディレクトリ。
**道具は規約をコード化するだけ**で規約を変えない。コード化する手順の SSOT は `docs/protocol.md`。

## 道具（C3 = bd un-4nm / un-3yc・実装済み）

admin / worker の手作業（protocol.md §1/§2/§5）を 1 コマンド化する道具群。**spawn/gate-args/selftest-args/cleanup には `--dry-run`**（実行するはずのコマンド列を arg-echo・実 spawn はコスト大）。

- `scribe-spawn.sh [opts] <bd-id>` — spawn ヘルパー（worker モード）。`bd show` で実在を事前検証（**fail-loud**＝cld-spawn の不正 id silent fallback を上流で塞ぐ）→ `git worktree add`（`spawn/<id>-HHMMSS`）→ task prompt 生成（契約=bd description 参照・cell-quality WF 起動・bdw 規律・禁止事項）→ `cld-spawn --bd-id <id> --model opus` → monitor（window ID `@N` 参照＝dotted id の `-t` 衝突回避）。
  - `--consult` で **consult モード**に分岐（`docs/role-context-spec.md` §2.3 / scribe-design.md §14 の契約どおり）: consult は anchor 同居・read-only セッションなので **worktree も worker prompt も `--bd-id` も作らない**。anchor で `cld-spawn --model <fable 既定・不可時 opus> --env-file <SCRIBE_ROLE=consult> "<consult テンプレ>"`（設計議論/grill 専用・記憶系のみ write・サマリ保存義務）を出す。bd id は consult では任意の議題参照（read-only な実在検証のみ）。`SCRIBE_ROLE=consult` は C2 の role 判定が最優先で読む side。
    - **env-file は anchor working tree の外（`/tmp` 配下を `mktemp`）に作り、`trap EXIT` + spawn 後 `rm` で必ず除去**する＝anchor リポ（admin の cwd）には一切作らない。read-only 契約の起動器が自身は anchor に artifact を残さない自浄。ファイル名も実体は `/tmp/scribe-consult-XXXXXX.env`（mktemp）なので、`.gitignore` への登録は不要（リポに生成され得ない）。
    - **モデル規約は worker と非対称**: worker は fable 厳禁（opus 必須・protocol.md §1）。consult は**既定 fable・利用不可時のみ opus へ loud fallback**（実起動時 preflight・`--model` 明示が優先・sc-9q6・SSOT = role-context-spec §2.3。consult は admin と同じ main-loop 系統ゆえ fable が許される唯一の役割。道具はこの規約を変えない）。
- `scribe-gate-args.sh [opts] --worktree <path> <bd-id>` — gate 支援（admin・protocol.md §5）。gate review 用 cell-quality args(JSON) を issue から合成し stdout へ。**`doPlan`/`doImplement`/`autoFix` は read-only 固定**（gate は一次監査・実装も autoFix もしない）。
- `scribe-selftest-args.sh [opts] --worktree <path> --self-test <cmd> <bd-id>` — worker 自己点検 支援（worker・protocol.md §2）。自己点検用 cell-quality args(JSON) を issue から合成し stdout へ。**`doImplement`/`doPlan`=false・`autoFix`=true 固定 + `selfTestCmd` 必須**（worker は実装済み・confirmed のみ gated 修正・fail-closed ゲート）。gate-args と対称だが**責務だけ非対称**（gate=read-only / 自己点検=gated autoFix）。手作業で args を組まず道具を使うこと。
- `scribe-cleanup.sh [opts] <bd-id>` — cleanup。merge 後の worktree remove / branch 安全削除 / window kill（`@N` 参照）/ `bd dolt push` リマインドのチェックリスト。**破壊操作は確認プロンプト付き**（`--yes` で一括承認）・**force 系（force 削除 / hard reset / tmux サーバ破壊）は使わない**。
- `scribe-origin-guard.sh {capture,verify,restore} --worktree <path>` — origin 健全性ガード（bd un-1n1）。worktree は anchor と `.git/config`（remotes）を共有し、worker が origin を mutate すると anchor+全 worktree の origin が壊れる（un-v5x 実害）。`capture`（spawn 時・`scribe-spawn.sh` が `git worktree add` 直後に自動実行）で canonical origin URL を **per-worktree marker**（`.git/worktrees/<name>/scribe-origin.marker`＝共有 config と別物・working tree 外）へ捕捉し、`verify`（admin gate funnel §5・push 前）で現在 origin と照合（健全=exit 0 / 汚染=exit 非0・canonical URL を stdout）。`verify --restore` or `restore` で marker から復元。
- `scribe-inject.sh {verify,send,marker,busy-check}` — admin 操舵注入の**送達確認**ヘルパ（sc-6vj）+ **送信前 busy-check gate**（sc-6mtm）。2026-07-08 に操舵注入の送達失敗が 2 度実発生した（bracketed-paste 滞留・`-l` 逐字滞留を pane 目視で誤『送信済み』判定）ことへの対策。`verify` は capture-pane 出力（`--capture-file`/stdin）と注入 payload の `--marker` を受け、CC TUI 入力ボックス *内部* に marker が残るか（=未送達か）を機械判定する（送信済みで transcript に echo された自分のテキストとは弁別）。`send --target <pane> (--file|--text)` は **送信前に busy-check gate を通り**（入力欄が非空 or 特定不能なら 1 キーも送らず defer）、通過時のみ send-keys -l + Enter → capture → verify し、残留なら Enter を追送して回復、retry 尽きても残留なら **fail-loud**。`busy-check (--target|--capture-file)` はその gate 単体（push 可否の事前判定）。終了コード: 0=DELIVERED/IDLE / 3=RESIDUAL / 4=INCONCLUSIVE（入力欄特定不能＝保守的 fail-loud） / **5=DEFERRED（送っていない＝gate が止めた・再送せず bead/mailbox へ）**。`--clear-first` 等の入力欄 wipe（C-u 相当）は **die で機械拒否**（no-push 原則）。tmux バイナリは `SCRIBE_TMUX` で差し替え可（テスト seam・live server 非依存で回帰テスト）。設計判断（tmux の after-send-keys hook / global pipe-pane は見送り）・**co-submit を confirm-receipt/verify が検知できない限界**の rationale は script ヘッダが SSOT（運用規約は `docs/protocol.md` §6「no-push 原則」）。
- `scribe-rebrief-fetch.sh` — **rebrief cycle の機械層 fetch engine**（bd sc-8eyw・`/scribe:rebrief` の DATA 供給元）。admin respawn / compaction 後の復元に要る生データを **read-only** で emit する（一切 write しない）。担うのは generic core 4 項目のみ: ①WM 主張↔bd 現在値 diff ②orphan WM surface ③auto-compact 強制回復 mode の**判定**（marker write / hook 新設は scope 外）④consumed 化対象の特定（consume＝`.md`→`.consumed.md` の mv は skill 層の責務）。**台帳 prefix は `.beads/metadata.json` の `dolt_database` を walk-up で動的解決**する（`hooks/lib/mailbox-common.sh` の `mbx_resolve_self_db` を再利用＝台帳解決を二重実装しない）ので per-project 設定が要らない。orchestrator overlay（STALE 停滞 scan / clean-state-probe の GREEN gate / gate-pending 列挙）は**持たない**＝欲しい層が自分で compose する。識別・取得できない事象は一律 **fail-loud exit 1**（`dolt_database` 未確定 / anchor 不在・anchor 直下 metadata 不在＝祖先の別台帳へ束ねない / cc-session lib 不在・**版ずれ**（記号不在） / **bd read 失敗**（bd 不在・非 0 終了・timeout・JSON parse 不能） / **JSON parser（jq・python3）双方不在**＝粗い近似 parse へ degrade しない / **WM・WM dir が「在るが読めない」**＝権限・所有権ずれ）——「読めなかった」を「乖離なし・BD-COUNT=0」に化けさせないため（空台帳の `[]` は rc=0 で正当・WM dir の**不在**も正当ゆえ巻き込まない）。同じ規律の非 FATAL 版として、**current sid の WM が無いときは「乖離なし」を名乗らない**（`[DIFF-UNKNOWN]`＝突合していない）＋未 consumed WM を `[WM-CANDIDATE]`（mtime 降順）で復元候補に surface する——WM は sid で scope される一方 respawn / `/clear` は sid を変える（cc-session が verified と明記）ため「WM は在るのに exact 一致しない」は例外でなく既定経路で、ここで沈黙すると boot path で偽の全クリア＝作業放棄を招く（採用・consume の判断は skill 層＝fetch は read-only のまま）。前提: **cc-session(session plugin) の user-scope enable**（WM の path 解決・2 節 schema の SSOT がその lib）。
- `lib/scribe-lib.sh` — 共有ヘルパー（bd id 正規化・実在検証・命名・origin 健全性 capture/verify/restore）。source 専用。

テスト: `tests/scribe-rebrief-fetch.bats`（rebrief core の hermetic 検証。cc-session lib は fixture へ**供給**し SKIP させない＝SKIP-green 不受理。orphan 規則・SELF_PREFIX walk-up は変異注入で teeth 化済み）。`tests/scribe-tools.bats`（dry-run arg-echo を assert・実 spawn しない。bd は `tests/fixtures/bd-stub.sh` でスタブ）。`tests/scribe-inject.bats`（送達確認の残留検知を fixture capture で assert・tmux は `SCRIBE_TMUX` stub・**実 tmux 起動しない**）。worker 自身の fail-closed self-test は `selftest-<id>.local.sh`（untracked）。

## hook script（C2 = bd un-ck2 領分・本 cell は触らない）

- `hooks/session-start-role-inject.sh` — role 判定 + role 別 SessionStart 文脈注入。**C2 が実装**。`hooks/hooks.json` の SessionStart wire は本 script を `[ -x ]` ガード付きで参照済み（未実装でも no-op）。仕様 = `docs/role-context-spec.md`。

→ scribe-design.md §14 の「道具」3 本柱（spawn / gate-args / cleanup）に対応。worker 自己点検の `scribe-selftest-args.sh` は un-3yc で追加した 4 点目（protocol.md §2 の自己点検 args を 1 コマンド化）。`scribe-origin-guard.sh` は un-1n1 で追加した 5 点目（protocol.md §1 spawn 捕捉 / §5 push 前 verify で共有 .git/config の origin 汚染を防ぐ）。`scribe-inject.sh` は sc-6vj で追加した 6 点目（admin 操舵注入の送達確認＝送信後に入力欄へテキストが残留していないことの機械検知・残留時 fail-loud）。
